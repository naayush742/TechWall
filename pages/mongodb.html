01 — Overview

What exactly is MongoDB?

MongoDB is a document database — instead of splitting data across rigid tables, it stores each record as a self-contained, nested document that mirrors how developers already think in code.

  • 📄
    Document model

    Each record is a BSON (binary JSON) document — fields can nest, repeat, and vary between documents in the same collection.

  • 🧬
    Flexible schema

    Add a new field to one document without rewriting every other row or running a migration script.

  • 🪐
    Horizontal scaling (sharding)

    Spread a single database across many servers automatically as data grows, rather than buying one bigger machine.

  • ☁️
    Atlas (managed cloud)

    A fully managed version of MongoDB on AWS, Azure, or Google Cloud — including a free forever tier for learning.

Why this matters: a relational table forces every row into the same shape. A document doesn't.

🗂️ Relational table (fixed shape)

idnamepricewarranty_years
1Headphones89.992
2USB Cable9.99NULL

Every row must have the same columns — even when a field doesn't apply, like a warranty on a cable.

📦 Document (flexible shape)

{ name: "Headphones", price: 89.99, warranty: {years:2} }
{ name: "USB Cable", price: 9.99 }

The second document simply omits the warranty field entirely — no NULLs, no forced structure.

02 — History

From a failed cloud platform to the leading NoSQL database

MongoDB almost wasn't a database at all — it started as one piece of a much bigger, abandoned product.

2007

10gen is founded

Dwight Merriman, Eliot Horowitz, and Kevin Ryan — veterans of DoubleClick, where they'd hit the scaling limits of relational databases — found 10gen in New York City to build a new kind of platform.

2007–08

A database hidden inside a bigger product

10gen originally builds a full platform-as-a-service. The database layer underneath it all — nicknamed "MongoDB," from "humongous" — turns out to be the most valuable part.

2009

The pivot — and first public release

10gen abandons most of the original PaaS code to focus entirely on MongoDB, releasing it open source in February 2009. SourceForge becomes an early, very public adopter.

2011–12

Enterprise adoption begins

Early Web 2.0 users like Craigslist and Foursquare give way to enterprise customers including Disney and eBay, proving MongoDB beyond the startup world.

2013

10gen renames itself MongoDB Inc.

The company drops its original name to align fully with its now-flagship product, reflecting where the real demand had landed.

2016

MongoDB Atlas launches

A fully managed, cloud-hosted version of MongoDB ships — removing the need to install or operate the database manually. By 2024 it accounts for over 70% of company revenue.

2017

MongoDB goes public

MongoDB files for IPO in September 2017 and lists on NASDAQ that October — completing the journey from failed PaaS startup to public company.

2018–26

ACID transactions, vector search & AI

MongoDB adds multi-document ACID transactions to satisfy enterprise needs, then Atlas Vector Search to support AI and RAG applications — broadening far beyond its original NoSQL identity.

03 — Use cases

Who actually builds on MongoDB

MongoDB tends to show up wherever data doesn't fit neatly into rows and columns, or where the shape of that data keeps changing.

Mobile & web apps

User profiles & content

App data — profiles, posts, settings — naturally fits the document model since each user's data can look slightly different.

E-commerce catalogs

Products with wildly different attributes

A t-shirt has size and color; a laptop has RAM and screen size. Documents handle that variety without sparse, NULL-filled tables.

IoT & real-time data

High-volume sensor streams

Sharding lets MongoDB absorb huge, constant write loads from connected devices by spreading them across many servers.

AI & RAG applications

Vector search for AI retrieval

Atlas Vector Search stores embeddings alongside regular data, powering AI chat features without a separate vector database.

Content management

Articles, comments & media

Nested comment threads and varying article layouts map naturally onto nested documents instead of multiple joined tables.

Learning

Students & rapid prototyping

The free Atlas tier and JSON-like syntax make MongoDB a fast, low-friction way to start building without designing a schema first.

04 — How it works

How MongoDB scales across many servers

Instead of one giant machine, MongoDB can split a database into "shards" — smaller pieces spread across multiple servers that work together.

STEP 1

Insert a document

insertOne() adds a JSON-like document directly — no table to define first.

STEP 2

Query with filters

find() matches documents by field values, even nested or array fields.

STEP 3

Aggregate & transform

The aggregation pipeline reshapes, groups, and summarizes data in a single chained query.

STEP 4

Scale with sharding

As data grows, Atlas automatically distributes it across shards — no manual repartitioning.

05 — Pricing

Free to start, pay-as-you-grow in the cloud

MongoDB Community Server is free forever to self-host — Atlas adds a fully managed cloud option with its own free tier.

Community Server
$0
Free, source-available, and self-hosted. The full database engine — run it on your own infrastructure.
Atlas Free (M0)
$0
Free forever managed cluster with 512MB storage — ideal for learning, prototypes, and small projects.
Atlas Dedicated
$57/mo+
Production-grade clusters (M10 and up) with backups, multi-region support, and advanced security.

Atlas Flex sits between Free and Dedicated at $8–$30/month with usage-based scaling. Pricing varies by cloud provider, region, and configuration — check MongoDB's official pricing page for current numbers.
